# HexSchool 哞職順利

Posted by 2021/3/25

前言

我是 WJ，目前任職於南部某百貨的資訊部一年半時間，工作內容除了網站前端視覺呈現和 APP 的維護外，還包括了些微後端的資料庫、API 建置及報表開發。也另外在下班後接了兩間廠商的遠端案件賺賺外快，內容則是切出設計稿及製作簡易的報名網站。

在這篇文章中，我將會提到我的就學背景、在六角學院的學習過程和面試，最後則是目前工作上及接案的所有事情。如果有興趣再看下去囉。

學習階段

我畢業於高雄海洋科技大學的資訊管理系，所以和其他六角的學生比起來，算是略有一些優勢，不過學校的內容囊括了很多項目，前端的部分有基礎的 HTML、CSS、jQuery， 後端的部分則是 PHP 還有資料庫 mysql，而除了上述以外，其他還包含了網路管理、Linux、資料結構、演算法.....等，其實一開始都不在意，但實際就業之後發現，這真的超重要的啦！！

而在約莫大三的時候開始想著未來到底要做甚麼工作，在此時這碰巧看見了六角學院的促銷活動，就把有興趣的課程都囊括下來

因為有些簡單的基礎，所以課程都上的 2266 XD ， 最一開始則是從 Bootstrap 4 這門課程開始，在上課對於 Bootstrap 的基礎認知為，到官網上把示範程式碼複製到編輯器中再改字， 而外觀樣式顏色甚麼的再自己苦刻 CSS，上完才發現原來他有這麼多好用的 Utility ，甚麼 padding margin 也不用慢慢手寫，瞬間發現了新大陸，也意識到了懂得如何閱讀文件的重要，大部分的 plugin 都是只有簡單的範例，精華都在於文件中。

也在這門課中認識了 SCSS 如此好用的工具，建議一同上 Bootstrap 的同學於這門課之餘搭配洧杰老師的 Sass 實戰全攻略服用，更能夠全面性的掌握 Bootstrap，大多數的職場中，也會使用 Bootstrap 當作框架進而修改。 也能更理解如何去拆解網頁中的每個元件，以及一些 HTML CSS 設計撰寫上的邏輯。

而在這門課程中，我更深入了解所謂的 Layout Grid 網格的設計、Breakpoints 斷點的應用，這些再往後 RWD 網頁應用上都是不可或缺的。

再來挑戰了 Vue 出一個電商網站，相信對於非本科系的同學來說，接觸 JS 絕對會非常有感，不像 HTML CSS 般好理解，但課程中在每個環節都有不同的練習題，都是老師們 用心透過每個章節的內容編排，只要按部就班，有問題就馬上在提問區或 Slack 提出，都會有助教及熱心的學長姊回覆，有時候不懂就可以透過其他人的回答進一步分解成自己的邏輯觀念，這也是我最喜歡六角的地方。

透過所學的內容完成每個作業除了更能夠理解程式，也可以從中獲得滿滿的成就感，讓你更喜歡程式，畢竟這是吃飯的工具，未來要長時間和它相處在一起，但也不是學好了 Vue 就可以完全不學原生 JS，未來也可能有更方便更符合潮流的框架 。

面試階段

我算是非常幸運的，只面試了兩間都有拿到 Offer，同時也建議同學們，在投稿履歷及面試前，都可以 Google 一下 該公司名子 ptt 這個關鍵字，或公司名子 面試，會有很多類似面試通這種經驗分享的網站， 可以初步了解面試的內容及可能會問到的問題，或是如果是南部高雄地區，可以問看看老師們有沒有甚麼建議，畢竟知己知彼百戰百勝~

第一間是位於高雄軟體園區內的科技公司，可能是面試 junior 的職缺，所以所問的問題都不艱深，若有好好上六角課程，都可以回答出來。

  • 為什麼想進入前端領域
  • 專題的內容及負責的工作
  • 請說說何謂 HTML、CSS、JS
  • 幾項 JS 的基本邏輯問題 (一年多前已經印象模糊了)
  • 最後則是目前公司是使用 angular，是否願意學習，公司也會有其他員工帶著

面談完之後沒有給我思考的機會，直接當場就給了我 Offer ，但想一想 angular 的學習曲線比 Vue 陡峭了許多，而在深入聊天的內容中也發現升遷及調薪的機會不多，索性就打退堂鼓了。

第二間也是目前的公司，因為大學工讀在同一間公司的關係，得知有職缺也當作經驗面試看看，前端上只有些許的導入 Vue ，其他都是使用 ASP.NET 及 jQuery 做開發，所以等於一進來就要開始學習後端程式，但公司可提供的資源較多，所以我最後選擇近來闖闖。

面試的題目是用考卷的方式，大概都是 JS 邏輯問題，還有一個最印相深刻，也可能是大家會覺得學過但卻寫不出來的手寫表格 HTML，當然考的就是 tr td 以及 rowspan 跟 colspan。現在的潮流多半使用 flex 進行排版， 但因為原有的網站都是 10 年前的架構了，所以 table 式的排版還是得會一點。

工作簡述&學習

這邊的工作環境，客戶都是公司內其他單位的員工，所以大多數的工作日常都是由使用者提出需求，再進行開發。

前端的部分主要是用 ASP.NET 維護著官網的前台以及後台的編輯，APP 的部分則也是使用 HTML + jQuery Mobile 再透過 Cordova 編譯，所以基本上來說上完六角的基本課程，也可以寫出一套 APP。

至於後端的部分則是從資料庫中取出需要的資料，再透過 C# 整理成 json 格式，讓前端的 ajax 可以呼叫，自給自足，自己的 API 自己開，就可以避免下面的狀況 ...

進階的部分像是使用 node.js 撰寫一個 Windows 服務，常駐於背景，當有人上傳了圖片到某個資料夾，就觸發 node 的 image plugin 對該圖片進行壓縮。

很多人在練習的時候都覺得，不知道從哪裡下手，但就可以想想手邊有什麼需求是可以透過程式解決，就是最好的練習。

而我上班的一天，是要從 0 開始學習 ASP.NET 和 C#， 聽起來非常可怕，但實際上 ASP.NET 的前端內容就是 HTML 再加上了些許的伺服器語言，而後端的 C# 其實也是和 JS 一樣有著 for 迴圈、if else，只是寫法不一樣，開始這樣催眠著自己 。

雖然這樣在職場學習的機會可能並不常見，就我而言好處就是可以站在前人的肩膀上，看著已經運行的程式碼來學習邏輯，並去理解這麼做的原因是什麼，要是自己究竟會怎麼寫，看著別人的 code 學習也是一種。

工作流程

因為我所工作的地方非資訊公司，所以流程上可能會和其他外面的流程不太一樣，所以這邊就僅供參考。

需求訪談
在使用者提出需求後，需要會議進行訪談，由使用製作相關需求 ptt ，在會議中若有不可行或需修改之處則提出 。
時程評估
在與 User 確認功能無誤後，需要規劃從何著手，先做前端或者後端資料先出，再估計完成時數彙報主管及 User 。
開發
按需求及規劃流程進行程式開發。
內部測試
使用測試單元或內部會議進行壓力測試，針對 BUG 進行修改。
User 驗測
透過 User 實際操作是否有功能或 BUG 做調整。
正式上線
User 驗測完畢於正式機上線，並於官網露出。

以下為範例：

需求訪談:
使用者提出前台網站示意圖及相關欄位，並告知需有後台上架圖文
時程評估:
前台使用 uikit 當作框架排版，後台設計資料庫欄位、建置上傳圖文功能，加上內部壓力測試共 1 週。
開發:
依照評估規劃進行開發，含問題除錯。
內部測試:
供內部課員壓力測試，並告知可以修改之處進行優化。
User 驗測:
依照評估規劃進行開發，含問題除錯。
正式上線:
User 驗測完畢於正式機上線。

若為資訊公司應該會有前後端的分離，前後端應再開一次會議進行協調時程及產出的內容：後端產出 API ，或前端於期限內交付程式碼予後端套上。

不過目前我則是單打獨鬥樣樣自己來，所以就不會有中間打架的困擾。

程式除錯&解決問題

在開發的過程中，一定(有機率)會遇到沒有碰過的錯誤或者是程式寫不出來，這邊也分享一下我的辦法：

  • HTML CSS 問題：擅用 Google ，而關鍵字又以英文輸入會更容易找到解法，通常都會跑出 stackoverflow
  • JS 問題：直接將所報錯的內容以 Google 搜尋，雖然問題並不是一模一樣，但通常都可以在回答看出端倪，像是不是型態錯誤
  • Plugin 問題：我會建議直接到該 Plugin 的 github page issue 搜尋有沒有遇到一樣問題的人，說不定官方有相對的解答。沒有的話在以 Google 搜尋報錯的內容及 Plugin 名稱。

另外就是擅用前輩 (具有善意的前輩)：有開發經驗的前輩理當吃過的蟲比你還多，說不定這隻蟲他也吃過，再不來就是一起討論解決方法，也不是只有一種寫法可以完成程式，所以有時候也需要大家的腦力激盪。

再不然就貼在六角學院的 Slack 吧 .... ，大家都是很熱心的。

經驗分享

1. 是的，寫好註解很重要

相信我，沒人在 junior 就能寫得出一手好 Code，所以在程式複雜之處寫上註解也是一個好習慣，除了別人好看之外，隨著資歷越久經手的專案更多，但難免有上線後需要維護的時候，總不希望在回頭維護時發現，哇靠，這誰寫的爛 Code。

可以參考這篇文章： 注释那些事儿 - 前端代码质量系列文章（一）

題外話，曾經看過一段註解寫著：

// 這個功能我寫不出來，但他應該暫時用不到

2. 無論改了多小的東西都要測試在上線

這也是親身經歷，上線時 User 提出了一個小問題，自信滿滿的我直接上正式機編輯，儲存收工下班。隔天，雪花般的電話及信件。

所以說，在微小的修改，最好還是要在上線前測試，若遇到金流問題，可不是 Say Sorry 可以解決的。
